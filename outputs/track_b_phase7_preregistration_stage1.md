# Track B Phase 7 — DLC SuperAnimal-Quadruped keypoint-anchored eye cropping

## Stage 1 pre-registration — principles + meta-rules

**Frozen 2026-05-09 BEFORE any DLC inference on the 34 RME clips.** Stage
1 locks everything that can be locked without RME observation. Stage 2 (a
separate amendment) fills in concrete threshold values via locked
meta-rules, after Step 2 (Wikimedia parity) + Step 1.5 (DLC inference on
RME clips, no crops yet) produce the observations that calibrate them.

This staged pattern was proposed by user in response to CC validation:
*"the right pattern is two-stage pre-reg ... lock the rule for setting
thresholds before observation, lock the concrete value after
observation."*

## What Phase 7 tests

Phase 6(b) closed with DISTRIBUTED_FAIL (AUC 0.4689; median IoU 0.165
vs Phase 5 manual boxes; loss_concentration_pct = 6.2%, locked routing
to DLC). Mechanism: positional miscalibration scaling with continuous
horse profile angle.

Phase 7 tests whether **per-frame DLC SuperAnimal-Quadruped eye
keypoints solve the localization problem** and whether keypoint-anchored
automated cropping preserves Phase 5's manual-cropping LOSO result
(AUC 0.7985).

## What Phase 7 is NOT

- Not a search for the best automation method. Phase 7 tests DLC
  specifically per Phase 6(b)'s locked routing.
- Not an opportunity to explore multiple keypoint→bbox geometries.
  Lock one (option A — DLC head-keypoint proxy bbox); if it fails,
  that's data for Phase 8.
- Not an opportunity to retroactively adjust gates. Locked thresholds
  are locked.

The Phase 6(b) failure was the discipline pattern's cleanest empirical
falsification so far. Phase 7 follows the same pattern. If DLC also
fails, that's another clean data point about which automation strategies
don't work.

## Section 4 — Eye-selection rule (LOCKED)

Per CC's verified per-clip side signal (25 clips manual eye box
right-of-face-center, 9 clips left-of-face-center across 34 RME clips),
rule (b) is implementable.

**Locked rule** (per-clip):

1. **Pre-DLC-inference: derive per-clip side assignment** mechanically
   from `outputs/eye_boxes_phase5a.json` + `outputs/phase6b_position_param.json`
   (via `tools/phase7_derive_side_assignment.py`):
   - For each clip, compute `offset = (eye_cx - face_cx) / face_width`
     where `eye_cx` = manual middle-keyframe eye box centroid x and
     `face_cx, face_width` = Phase 6(b)-cached face bbox.
   - **If `offset > +0.05`** → side assignment = `right_of_face_center`
     (horse pointing camera-left, anatomical RIGHT eye visible) →
     **target DLC keypoint = `right_eye`**.
   - **If `offset < −0.05`** → side assignment = `left_of_face_center`
     (horse pointing camera-right, anatomical LEFT eye visible) →
     **target DLC keypoint = `left_eye`**.
   - **If `|offset| ≤ 0.05`** → side assignment = `ambiguous` →
     **target = higher-confidence keypoint at inference time**
     (rule-(a) fallback, locked here).

   Margin threshold of 0.05 is locked at this round number, not tuned.
   **Verified ambiguous count at lock time**: 6 of 34 clips
   (18%) fall in the |offset| < 0.05 zone. Threshold sensitivity:
   0.03 → 2; 0.04 → 4; 0.05 → 6; 0.06 → 8; 0.07 → 10. Path-A
   (margin=0.05 with per-clip-locked fallback per item 2 below) is
   chosen over Path-B (margin=0.03, fewer fallback clips) because the
   smaller offsets between 0.03 and 0.05 likely reflect near-frontal
   shots or annotation framing artifacts, not principled side
   identification — forcing a side based on those small offsets is
   more misleading than treating them as ambiguous and letting DLC's
   own confidence drive the per-clip lock.

   **Diagnostic flag** (locked here, surfaced in §10): two of the 6
   ambiguous-zone clips are diagnostically meaningful — `action_S5.mp4_5_`
   (offset −0.028, also V3_NEWLY_LOST and Sensitivity-1 swap pair clip)
   and `background_S3.mp4_3_` (offset −0.046, also V3_NEWLY_LOST). These
   clips' DLC outcomes must be reported with explicit notation that the
   per-clip-locked-eye fallback rule was applied, not the principled
   side-assignment rule. Audit must not conflate "fallback rule applied"
   with "DLC failed on this clip."

2. **At DLC inference time**:
   - If clip side = `right_of_face_center` or `left_of_face_center`:
     **per-frame** decision — use the assigned eye keypoint UNLESS its
     confidence < §7-threshold AND the opposite eye keypoint has confidence
     > 0.7. In that frame-level fallback case, swap to the higher-confidence
     eye for that frame only and log the swap in the manifest.
   - If clip side = `ambiguous`: **per-clip lock**, not per-frame. Compute
     the majority-confident eye across the clip's confident-frame detections
     (frames where at least one eye keypoint clears §7-threshold), pick the
     eye that is more often the higher-confidence one, lock that eye for
     all frames in that clip. This prevents per-frame oscillation between
     left/right_eye that would cause crop jitter on near-frontal shots.
     The locked eye is recorded in the manifest as `clip_locked_eye:
     "right_eye" | "left_eye"`.

3. **Hash-locked artifact**: `outputs/phase7_eye_side_assignment.json`
   computed by `tools/phase7_derive_side_assignment.py` BEFORE any
   DLC inference on the 34 RME clips. The output schema:
   ```json
   {
     "tool_hash": "...",
     "margin_threshold": 0.05,
     "per_clip": {
       "<filename>": {
         "offset": <float>,
         "side": "right_of_face_center" | "left_of_face_center" | "ambiguous",
         "target_eye_keypoint": "right_eye" | "left_eye" | "higher_conf_per_frame"
       },
       ...
     },
     "summary": {"right": <n>, "left": <n>, "ambiguous": <n>}
   }
   ```

## Section 5 — Keypoint→bbox geometry (LOCKED, option A)

DLC produces 2D keypoints, not bboxes. Phase 6(b)'s `rel_w/rel_h` ratios
were *face-bbox-relative*. Bridge: derive a face bbox proxy from DLC's
own head keypoints (single-model, no second detector).

**Locked rule** (per-frame):

**Terminology lock**: throughout §5 and §6, "**confident keypoint**"
means "keypoint with confidence ≥ §7-threshold." The §7-threshold's
concrete value is set in Stage 2 via the locked meta-rule there. In
Stage 1, "confident" is bound to the §7 meta-rule, not to any
provisional 0.5 value.

1. Compute `head_keypoint_bbox`: minimum axis-aligned rectangle
   enclosing all head keypoints from §2's locked 9-keypoint set
   that are **confident** (per the terminology lock above).

2. **Minimum-confident-keypoints gate**: if **fewer than 6 of 9** head
   keypoints are confident in a frame, the head-bbox proxy is
   considered untrustworthy. **Fallback**: option C —
   eye-keypoint position + locked absolute pixel dimensions
   (`abs_eye_w_px`, `abs_eye_h_px` derived from Phase 5 manual boxes
   median absolute dimensions, computed in stage 2).

   This locks one degree of freedom that CC flagged: at ~9% per-keypoint
   failure rate, a frame can have 4-5 confident head keypoints, and
   the bbox enclosing 4 vs 9 differs by 30-50% in dimensions. The
   ≥6 threshold prevents silent geometry drift between confident-rich
   and confident-poor frames.

3. **If proxy bbox is trustworthy** (≥6 confident head keypoints):
   - Center the eye box at the target DLC eye keypoint coordinate
     (per §4 rule).
   - Eye-box dimensions = `rel_w × proxy_bbox_width`,
     `rel_h × proxy_bbox_height` where `rel_w = 0.1419`, `rel_h = 0.0596`
     (Phase 6(b) hash-locked at `d1521a35c2de...`).

4. **Apply margin** m=15% (matches Phase 5 primary), square-pad,
   resize 224×224. Identical post-processing to Phase 5 v3 / Phase 6(b).

This preserves the "DLC alone solves localization" framing
(no second detector), reuses the hash-locked Phase 6(b) ratios,
and degrades gracefully via the locked fallback.

## Section 6 — Frame-level DLC failure handling (META-RULE LOCKED, values in stage 2)

**Locked structure**:

1. Per-frame attempt: run DLC inference on every native frame.
2. Single-frame eye-keypoint failure (target keypoint conf below
   threshold AND fallback rule from §4 doesn't recover) →
   **interpolate eye keypoint linearly** between nearest temporally-preceding
   and following frames where the target keypoint cleared threshold.
3. Single-frame head-keypoint-proxy failure (<6 of 9 head keypoints
   confident) → use eye-keypoint + absolute fallback dimensions
   per §5 step 2.
4. Clip-level threshold: if **> X%** of frames have eye-keypoint
   failure → fall back to **single-keypoint applied across the
   entire clip** (median target eye keypoint position; locked
   absolute fallback dimensions). Manifest logs `fallback_mode:
   "single_keypoint"`.
5. If **> Y%** of frames have eye-keypoint failure → drop clip from
   LOSO. Manifest records `status: "fail"` with reason.

**X% and Y% are locked in stage 2** (after observing DLC eye-keypoint
confidence distribution on RME clips). Meta-rule for setting them:
- Default: X = 25% and Y = 50% (Phase 6(b) values), unless DLC
  failure characteristics differ substantially.
- "Substantially different" = if observed failure rate on RME clips
  is more than 2× Phase 0's ~9%, X and Y need adjustment to keep the
  fallback regime uncrowded.

## Section 7 — Eye-keypoint confidence threshold (META-RULE LOCKED, value in stage 2)

The "confidence threshold" used in §4 fallback rule and §6 failure
handling is itself derived from observation.

**Meta-rule** (locked):
- Threshold = `max(0.5, p25_pooled_observed)`.
- **`p25_pooled_observed` is locked as the 25th percentile of the
  pooled distribution of target-eye-keypoint confidences across all
  34 RME clips × all native frames where the target eye keypoint
  was emitted by DLC** (i.e., flatten clips × frames into a single
  vector, then take p25). This is **NOT** per-clip p25 averaged
  across clips — those two computations give different numbers when
  some clips have systematically lower confidence (e.g., dim-lit or
  motion-blurred clips drag down their per-clip p25 even if their
  median is fine), and the pooled version is the data-grounded
  global threshold.
- The 0.5 floor matches Phase 0's natural break in the distribution
  (median 0.84-0.87, ~9% of frames < 0.5).
- The pooled p25 pegs the threshold to actual data; if DLC is
  uniformly very confident on RME clips, the threshold is the floor;
  if confidence is bimodal or has a long tail, the threshold rises.

Concrete `p25_pooled_observed` value locked in stage 2 amendment after
Step 1.5 observations.

## Section 9 — DLC-specific failure-mode routing (LOCKED)

CC's correction: Phase 6(b)'s OE-concentration rule was face-bbox-position
specific and doesn't transfer to DLC's failure mechanisms. DLC failures
are categorized along **two diagnostic axes**, not OE-concentration.

**Diagnostics** (computed in `tools/phase7_diagnostic.py`):

- `mean_keypoint_conf_on_failed_clips`: mean target-eye-keypoint
  confidence on clips that are FBB_NEWLY_LOST relative to Phase 5
  primary.
- `median_iou_on_failed_clips`: median IoU vs Phase 5 manual boxes
  on the same FBB_NEWLY_LOST set.

**Routing matrix**:

| AUC verdict | Mechanism diagnostic | Verdict | Next |
|---|---|---|---|
| Both gates clear | (n/a) | **PASS** — viable Phase 8 scaling baseline | Defer custom YOLO unless N ≥ 200 commits |
| AUC > 0.7985 AND paired p < 0.05 | (n/a) | **OUTPERFORM_PHASE_5** | Strong finding; investigate which clips DLC handles better than human annotation |
| G1 passes, G2 (AUC) fails | IoU-on-losses < 0.3 OR mean conf < 0.5 | **NON_INFERIORITY_FAIL_LOCALIZATION** | Geometry/side-assignment review; possibly orientation-aware Phase 8a |
| G1 passes, G2 (AUC) fails | IoU-on-losses ≥ 0.5 AND mean conf ≥ 0.7 | **NON_INFERIORITY_FAIL_SCOPE** | Architecture/scope issue, not localization. Larger margin (m=80%) test or DLC-then-different-feature-extractor |
| G1 fails | mean conf < 0.5 on failed clips | **CONFIDENCE_BOTTLENECK_FAIL** | Custom YOLO horse-eye detector (Phase 8b, 2-3 day investment). DLC sees "no eye" reliably enough to need a detector trained specifically for horse eyes. |
| G1 fails | mean conf ≥ 0.7 AND IoU < 0.3 | **CONFIDENT_MISPLACEMENT_FAIL** | Geometry/side-assignment review first; if that doesn't recover, orientation-aware Phase 8a. |
| G1 fails | mean conf ≥ 0.7 AND IoU ≥ 0.5 | **ON_EYE_WRONG_PREDICTION_FAIL** | Architecture/scope issue: localization works but the V-JEPA-2 + LR signal isn't extractable from this crop scale. Larger margin test, or different feature-extractor. NOT custom YOLO (which would also crop tightly). |
| G1 fails | Mixed (no diagnostic crosses threshold cleanly) | **AMBIGUOUS_FAIL** | Run BOTH custom YOLO AND geometry review; let empirical comparison disambiguate. |

**OE-concentration on the 4 pre-named clips** (bg_S5_10, action_S9_7,
action_S9_4, bg_S1_12) is reported as a **secondary diagnostic** (does
DLC handle the 4 OE clips Phase 6(b) face-bbox failed on?), but does
NOT enter the routing decision. It informs Phase 8 design if escalation
is needed.

## Section 11 — Anti-patterns (LOCKED)

1. **No mid-run parameter tuning.** All parameters locked in stage 1
   + stage 2 (via meta-rules). Re-run only if execution literally
   fails (e.g., file format incompatibility), not if results are
   uninteresting.
2. **No alternative head-keypoint sets.** Locked at 9 keypoints in
   §2; not adjustable.
3. **No alternative eye-keypoint position weighting.** Use the target
   keypoint coordinate as-is, not a weighted average of left+right_eye.
4. **No retroactive failure-mode definition.** The dual-diagnostic
   routing matrix in §9 is locked; routing decisions follow it
   mechanically.
5. **No removing clips with low DLC confidence from the LOSO.** They
   stay in (subject to the locked failure-handling rules); their
   behavior IS the diagnostic.
6. **No iterating on DLC version mid-phase.** rc13 is locked. If rc14
   ships, the discipline-pattern reproduction event clause (§1) fires:
   re-run the entire chain as a separate event, not silently upgrade.
7. **Don't celebrate G3 (IoU) ≥ 0.6 alone as success.** G3 is reportable,
   not gating. Phase 6(b) demonstrated IoU and AUC can diverge.
